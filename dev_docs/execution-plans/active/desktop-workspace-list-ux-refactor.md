# 桌面工作区列表与操作层级 UIUX 改进计划

## 背景

当前项目与任务工作区的主要问题不是单个控件样式，而是“操作、筛选、创建、浏览、详情”在同一视图层级中纵向平铺，导致核心数据列表被挤压，右侧详情区利用率不足，整体观感偏向中后台表单拼接。

现状中需要重点改进的区域包括：

- 项目注册表左栏常驻展示“创建项目”长表单，创建动作长期占据项目浏览空间。
- 任务工作区中筛选表单、创建任务表单和任务列表纵向同权堆叠，任务列表首屏信息密度过低。
- 筛选器以大块表单呈现，视觉权重高于实际使用频率。
- 列表项偏卡片化，依赖容器、圆角和块状背景表达层级，和桌面生产力工具应有的高密度列表浏览不匹配。

本计划需要同时遵守以下边界：

- `UI_DESIGN.md` 是交互结构真相源；项目页仍应保持“左侧项目注册表 + 右侧当前项目工作区”的父壳层，不移除项目注册表。
- `UI_STYLE.md` 是视觉 token 真相源；列表、筛选器、按钮和状态必须复用既有灰阶、紧凑高度、细边界和 Selected 状态语法。
- `frontend-skill` 对 app UI 的约束适用于本计划：Linear-style restraint、cardless lists、minimal chrome、高信息密度、操作入口轻量化。
- 前端用户可见文案必须走 `vue-i18n`，任何新增或调整文案需要同步更新 `en` 与 `zh-CN`。

## 目标

本次改进目标是把项目与任务工作区从“表单堆叠页面”收敛为“浏览优先、操作就地展开、详情承载编辑”的桌面工作台。

具体目标：

- 项目注册表变为纯浏览列表：标题、数量、轻量创建入口和项目行，创建项目不再常驻占位。
- 任务工作区变为列表优先：顶部紧凑工具栏承载版本筛选、状态筛选和创建入口，下方空间优先留给任务列表。
- 创建任务从常驻表单改为上下文内操作：优先使用右侧详情区的创建模式；如实现复杂度或空间不合适，再使用当前上下文 Drawer。
- 筛选器从纵向表单改为 compact toolbar：任务页默认绑定当前项目，仅保留版本和状态筛选。
- 项目与任务列表项去卡片化：使用 row、细分割线、轻微 hover、明确 selected，提升单屏信息密度。
- 保持键盘与焦点语义清晰：创建入口、筛选器、列表行、详情/Drawer 关闭都必须有明确 focus 行为。

## 非目标

- 不移除项目注册表，也不把项目切换迁移到全局侧边栏。
- 不新增一级导航，不把任务、版本、审批重新提升为全局入口。
- 不把普通创建流程做成 Modal；Modal 仍只用于阻断性或高风险确认。
- 不调整 Rust/Tauri 后端数据模型，除非实现过程中发现前端无法表达必要状态。
- 不引入新的品牌色、大圆角、大面积阴影或新的视觉 token。

## 方案

### 1. 项目注册表收敛

项目注册表继续作为项目工作区父壳层的左侧对象列表，但移除常驻创建项目表单。

建议结构：

- 顶部：`项目注册表` 标题、项目数量、一个紧凑 `+` 图标按钮。
- 中部：项目 row 列表，展示项目名称、slug、状态与必要元信息。
- 空态：无项目时显示短说明和创建入口，但仍不常驻展开完整表单。
- 创建入口：点击 `+` 后进入右侧工作区的创建项目模式，或打开当前上下文 Drawer。

注意事项：

- 项目 row 的 Selected 状态必须决定右侧当前项目工作区内容。
- Hover 只表达可探索，不改变项目上下文。
- 创建完成后应选中新项目并进入其 `overview`，焦点返回稳定上下文。

### 2. 任务工具栏与列表重构

任务页默认绑定当前项目，不再展示“当前项目”输入框或“全部项目”筛选。

建议结构：

- 顶部工具栏：版本筛选、状态筛选、可选搜索/过滤入口、任务数量、`+ 新建任务` 主动作。
- 列表区域：任务 row 贯穿列表列，单行或双行信息结构，优先展示标题、摘要、状态、优先级和必要时间/引用。
- 空态区分：无任务与当前筛选无结果需要使用不同文案和操作建议。
- 创建入口：点击 `+ 新建任务` 后右侧详情区切换为创建任务表单；若当前已有选中任务，应允许取消创建并回到原选中任务。

注意事项：

- 版本与状态筛选控件高度应按紧凑控件处理，避免 36px 以上表单堆叠。
- 筛选状态变更时应清除不再有效的任务选择，避免旧详情残留。
- 任务 row 的点击/键盘选择应更新 query 中的 `task`，保持可复制 URL 与恢复能力。

### 3. 右侧详情区承载创建与编辑

右侧 Detail/Inspector 不再只是“选中后展示”，也应承载当前上下文内的创建/编辑流程。

建议模式：

- 无选择：显示明确空态，说明可选择任务或新建任务。
- 查看任务：展示任务摘要、状态、详情 tab、备注、附件和活动。
- 创建任务：替换为空间稳定的创建表单，标题明确为“新建任务”，提供保存与取消。
- 编辑任务：保留在任务详情 overview 内，但减少卡片化分区，避免和创建模式视觉混淆。

注意事项：

- 创建/编辑表单字段需要沿用统一输入 token，不引入更厚重的卡片容器。
- 关闭创建模式后焦点应返回触发按钮或最近的任务行。
- 若后续改为 Drawer，Drawer 必须只承担当前任务上下文内的局部创建流程，不替代全局搜索或导航。

### 4. 列表视觉与状态语义统一

项目、任务、审批等列表应尽量共享同一套 row 语法。

建议基线：

- 默认 row 高度控制在 `36px ~ 40px`，必要时允许双行紧凑摘要。
- row 使用细分割线、弱边界或轻微灰阶变化区分，不使用大卡片阴影。
- Selected 使用“轻微底色 + 细边框 + 文本抬升”，不依赖彩色竖条。
- 状态只在小 badge、图标点或元信息中体现，不进行整行大面积染色。
- Hover 只提升一个视觉层级，不能和 Selected 长得相同。

### 5. 文案与可访问性

本改进会涉及新增入口、空态、取消创建、筛选说明等文案，必须同步更新 `src/i18n/messages.ts` 的 `en` 与 `zh-CN`。

需要验证：

- 所有新增按钮有可理解的文本或 `aria-label`。
- 任务详情 tab 仍保持真实 `tablist / tab / tabpanel` 语义。
- 创建模式、Drawer 或任何局部面板关闭后恢复焦点。
- 键盘可完成筛选切换、列表选择、创建入口触发和取消。

## 执行步骤

### 阶段一：结构收敛与入口调整

目标是先移除常驻长表单对浏览空间的挤压，但不大规模重写详情区。

任务：

- 将项目注册表中的创建项目长表单收敛为轻量入口。
- 将任务页顶部改成紧凑 toolbar，移除“当前项目”表单展示。
- 将创建任务常驻表单从列表列移出，先接入右侧详情区创建模式或临时 Drawer。
- 更新对应 i18n 文案。

交付判断：

- 项目列表与任务列表首屏空间明显扩大。
- 任务页在未创建时不再显示完整创建表单。
- `bun run build` 通过。

### 阶段二：列表 row 视觉统一

目标是把项目与任务列表从卡片感降级为高密度对象列表。

任务：

- 抽象或统一项目 row、任务 row 的基础 class 与状态语法。
- 收敛 row 圆角、边界、hover、selected、focus-visible 视觉。
- 检查状态 badge、优先级、数量胶囊是否符合小体量要求。
- 确保窄视口下列表仍可读，不产生多层挤压滚动。

交付判断：

- 项目和任务 row 的 Hover、Selected、Focus 语义可被肉眼区分。
- 单屏任务可见数量提升。
- 无大面积状态染色或卡片碎片化。

### 阶段三：详情区创建/编辑体验完善

目标是让右侧 Detail/Inspector 成为稳定的查看、创建和编辑承载区。

任务：

- 完善任务创建模式的保存、取消、错误、成功后选中逻辑。
- 评估项目创建是否也应进入右侧创建模式，或使用 Drawer。
- 调整任务详情 overview 内的编辑表单，减少卡片化和重复标题。
- 补齐焦点恢复与键盘路径。

交付判断：

- 新建任务流程不打断主列表上下文。
- 创建成功后新任务被选中并进入详情。
- 取消创建后返回原任务选择或无选择空态。

### 阶段四：验收与文档回写

目标是把实现结果沉淀回规范和计划状态。

任务：

- 根据最终交互选择，确认是否需要更新 `UI_DESIGN.md` 或 `UI_STYLE.md`。
- 更新本计划 TODO 状态与备注。
- 运行 `bun run build`。
- 如变更可见 UI，补充截图或录屏用于 PR 说明。

交付判断：

- 规范、实现和计划描述一致。
- 构建通过。
- 剩余问题被记录为后续 TODO，而不是隐含在实现中。

## 风险与关注点

- 如果创建项目和创建任务都进入右侧详情区，需要避免“详情查看”和“创建模式”视觉过于相似。
- 如果使用 Drawer，需要确保 Drawer 只承担当前上下文内局部操作，不扩展成新的全局导航模式。
- 列表去卡片化后，Selected 和 Hover 的可辨识度必须通过边界、文本对比和轻微底色保证，不能变成“看不出选中”。
- 筛选器收敛后，需要确保当前筛选条件仍然可见，避免为了克制牺牲可理解性。
