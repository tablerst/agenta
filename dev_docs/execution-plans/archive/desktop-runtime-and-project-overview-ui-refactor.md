# 桌面端 Runtime 与项目概览专项收敛重构

## 背景

当前桌面端已经完成项目工作区壳层与 Runtime 基础功能，但最近一轮预览暴露出一组明显的结构与视觉问题，且这些问题已经开始影响信息密度、可扫读性与操作心智：

- `项目概览` 视图仍然依赖 `glass-panel`、`metric-card`、`panel-section` 这类盒子式分组，形成明显的 dashboard-card mosaic。
- `项目概览` 左侧同时堆叠“项目摘要 / 元数据 / 快捷操作”，右侧又单独放置“编辑项目”，导致同一页面内存在过多二级标题和重复分区语义。
- `Runtime` 视图把 **MCP 主机生命周期与日志**、**本地路径与启动配置**、**远程同步状态与 outbox 操作** 强行缝在同一长页面里，造成认知任务混杂。
- `Runtime` 顶部五个状态块、启动配置区、日志区、路径区、同步区、outbox 区共同构成了“块状堆叠 + 整页滚动”的重型工作台，违背了本项目在 [UI_STYLE.md](D:/RustRover/agenta/UI_STYLE.md) 与 [UI_DESIGN.md](D:/RustRover/agenta/UI_DESIGN.md) 中强调的克制、高密度、低噪声原则。
- 当前文案仍带有明显的“页面介绍感”与“模块命名感”，例如“项目摘要”“快捷操作”“桌面运行时界面”等，会增加扫描跳点，不利于工具型工作流。

本专项的目标不是做一次泛化“美化”，而是把桌面端这两个最典型的高频视图收敛回 **Linear-style restraint + 工具型信息架构**：让结构先清楚，再让视觉安静。

## 方案

### 范围

- 本专项仅覆盖两个视图：
  - `/projects/:projectSlug/overview`
  - `/runtime`
- 若在实现中引入新的路由壳层、子路由或新的页面模式，需要同步更新 [UI_DESIGN.md](D:/RustRover/agenta/UI_DESIGN.md) 与 [UI_STYLE.md](D:/RustRover/agenta/UI_STYLE.md)。

### 设计裁决

#### 1. Runtime 与同步解耦，但不同步升格为一级导航

- `Runtime` 继续保留为左侧全局一级导航。
- `同步` 先不升格为新的全局一级入口，而是作为 `Runtime` 内部的二级区段或子路由承载。
- 推荐落地形式：
  - `/runtime/host`：主机生命周期、启动配置、路径、日志与恢复
  - `/runtime/sync`：远端信息、同步模式、检查点、outbox 与手动推拉操作
  - `/runtime`：兼容重定向到 `/runtime/host`
- 原因：当前同步能力仍紧贴桌面宿主运行时配置与本地副本状态，尚不足以抽象成独立全局工作域；但继续与主机控制共页会持续放大认知负担。

#### 2. 去卡片化是方向，但不是“把所有容器删光”

- 采纳 `frontend-skill` 的 `No cards by default` 与 `Apps -> Linear-style restraint` 原则。
- `glass-panel` 与 `metric-card` 不再作为默认分组手段。
- 只有在“容器本身就是交互对象”或“必须承担独立滚动/边界职责”时，才保留有边界的容器。
- 因此本专项的真实目标是：
  - 用对齐、留白、细分割线、字重和位置关系取代大多数盒子；
  - 保留少量必要容器，用来表达 Inspector、受控日志区、独立列表区，而不是为了“看起来像模块”。

#### 3. 输入控件要更安静，但不做全局下划线化

- 同意当前 `项目概览` 和 `Runtime` 表单输入框视觉存在感偏重，需要降低底色、边框和内阴影感。
- 不采纳“直接全面改成几乎无边框或仅下划线”的激进做法。
- 原因：
  - 本仓库已有明确的输入控件基线；
  - 产品是桌面运维/工作台，不是品牌展示页；
  - 过度轻量化会牺牲可扫描性与一致性。
- 落地原则：
  - 优先引入 **quiet field** 风格变体，而不是推翻现有 `control-input / control-textarea / control-select` 基线；
  - 若该变体会被多个视图复用，需同步进入设计文档。

#### 4. 动效只能服务结构切换，不做表演性级联

- 采纳“过渡应顺滑、不闪烁、不硬切”的目标。
- 不采纳大幅度、长时长、展示性很强的页面级联入场作为默认样式。
- 动效只允许服务以下场景：
  - 项目切换后工作区内容的稳定替换；
  - Runtime 子路由切换；
  - Inspector / 日志区域的轻量显隐与状态变更；
  - 表单聚焦、按钮按压、空态到有数据的细微过渡。
- 必须同时满足：
  - 无布局跳动；
  - 支持 reduced motion；
  - 不制造额外视觉噪声。

### 目标结构

#### A. 项目概览页

- 保持项目工作区壳层不变，继续遵守“左侧项目注册表 + 右侧当前项目工作区”模式。
- 在 `overview` 区段内部，改为 **主编辑列 + 轻量 Inspector 列**：
  - 主编辑列：项目标识、名称、说明、状态、保存动作；
  - Inspector 列：版本数、任务数、待审批数、默认版本、创建/更新时间等附带信息。
- 删除当前“快捷操作”模块，避免和项目工作区头部二级导航重复。
- `版本 / 任务 / 审批` 的进入方式以已有工作区二级导航为主；若保留概览内跳转入口，必须降级为低存在感的文本化链接或可点击指标，不再使用大块按钮列表。
- `保存项目` 作为页面唯一明确 Primary CTA。

#### B. Runtime 主机页

- 页头只保留主标题、当前状态、最近刷新时间，以及极少量元信息。
- 现有“五块状态卡”改为一条或两条紧凑 `metadata strip`：
  - 生命周期
  - endpoint
  - 日志去向
  - 配置来源
  - 同步状态摘要（仅作为跳转提示，不承载完整同步业务）
- 主区采用 **配置主列 + 日志 Inspector 列**：
  - 左列：启动配置、持久化选项、路径信息、失败恢复；
  - 右列：独立滚动日志面板，使用 `border-left` 和等宽字体强化工具属性。
- 日志区不再作为漂浮盒子，而是作为真正的次级上下文栏。

#### C. Runtime 同步页

- 远程同步从主机页独立出去，单独承载：
  - 远端类型、主机、数据库
  - 同步模式
  - push ack / pull checkpoint
  - pending outbox 数量
  - outbox 列表
  - 手动 backfill / push / pull
- 页面只允许一个主行动焦点。默认情况下：
  - `刷新`、`backfill`、`pull` 为次级动作；
  - `push` 是否成为主动作取决于当前业务优先级，但不应与主机页的 `启动主机` 同屏竞争。

### 视觉与文案原则

#### 视觉

- 继续使用黑白灰单色体系，不新增彩色主导区域。
- 大部分区块回到 plain layout，依赖：
  - 标题层级
  - 字重
  - 间距
  - 1px 分割线
  - 受控的独立滚动容器
- 降低以下元素的存在感：
  - `section-label` 的重复出现频率
  - `glass-panel` 的磨砂浮层感
  - `metric-card` 的独立边界感
  - 次要按钮的体积和对比度

#### 文案

- 优先使用 utility copy，而不是“模块命名型”标题。
- 优先删除或弱化下列文案类型：
  - “项目摘要”
  - “编辑项目”
  - “快捷操作”
  - “桌面运行时界面”
  - 过长的页面介绍句
- 保留必要的作用说明，但每段说明最多一句，且必须服务操作判断。
- 所有文案变更必须同步维护 `src/i18n/messages.ts` 的 `en` 与 `zh-CN`。

### 影响面

## 执行步骤

### Phase 1：信息架构与设计口径定稿

- 明确 `Runtime` 子路由方案与兼容跳转策略。
- 明确 `项目概览` 的主列 / Inspector 列职责边界。
- 明确哪些现有视觉模式要淘汰，哪些需要保留为低存在感容器。
- 若引入新的 `quiet field`、`metadata strip`、`runtime shell` 等模式，先补设计文档，再进入实现。

### Phase 2：项目概览页去卡片化与层级重排

- 移除 `overview` 内部对 `glass-panel`、`metric-card` 的默认依赖。
- 删除“快捷操作”块，收拢重复跳转入口。
- 将指标、默认版本、创建/更新时间收敛到轻量 Inspector。
- 重新组织表单层级，压缩重复 section label，仅保留保存动作作为主焦点。
- 视情况把指标做成低干扰可点击元素，但不得替代现有工作区主导航。

### Phase 3：Runtime 主机/同步拆分与日志 Inspector 化

- 为 `Runtime` 增加父壳层和二级导航。
- 主机页落地 metadata strip、配置主列、日志 Inspector、失败恢复区。
- 同步页承接 remote / checkpoint / outbox / 手动操作。
- 清理原先主机页内的同步业务块，避免同屏缝合。

### Phase 4：视觉细化、文案收编与动效收敛

- 降低按钮、输入控件、标题和空态的视觉噪声。
- 将低频说明改为工具型文案。
- 为路由切换、日志刷新、表单交互补齐克制过渡。
- 确保 reduced motion 下仍然可用且不闪烁。

### Phase 5：验证、截图回归与文档收尾

- 运行前端构建，确保 `vue-tsc` 与 Vite 构建通过。
- 对桌面常见视口进行手动验收：
  - 项目概览页
  - Runtime 主机页
  - Runtime 同步页
- 核对滚动语义：
  - 不把滚动责任回退到整页 `body`
  - 日志与 outbox 具备独立滚动
- 如新增了页面模式或输入变体，同步更新设计规范并关闭本专项。

## 当前结论

- 对 `项目概览` 与 `Runtime` 的批评方向总体成立，问题核心不是“丑”，而是 **职责缝合、层级过多、盒子依赖、操作主次失衡**。
- 额外补充的方案中，`去卡片化`、`文案工具化`、`日志 Inspector 化` 与 `Runtime / Sync 解耦` 应被采纳。
- 其中“输入全面下划线化”和“展示性较强的级联入场动画”不应直接采纳，应改造成适配本项目工作台语境的保守版本。
- 本专项完成后，桌面端这两个高频界面应回到“结构先清楚，视觉再安静”的基线，为后续更深的 Runtime / Sync 产品化演进留出空间。
