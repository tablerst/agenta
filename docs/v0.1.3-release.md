# v0.1.3 发布与打包说明

`v0.1.3` 是从 `v0.1.2` 之后收口的稳定性与可运维版本，重点覆盖 Desktop 自动同步、远端同步兼容、搜索索引增量构建、任务工作台可用性，以及数据库迁移 checksum 的跨平台一致性。

## Release Note

### 新增与改进

- Desktop 自动同步改为显式 opt-in：新增 `sync.auto.enabled`、`sync.auto.interval`、`sync.auto.batch_limit`、`sync.auto.startup_backfill`，默认关闭；启用后只在 Desktop 打开期间运行。
- 同步冲突处理更保守：本机内增加 sync run lock，远端 push 使用 CAS 语义，pull 能识别同版本不同来源的冲突；检测到冲突后自动同步暂停，不做静默覆盖。
- 远端 PostgreSQL sync schema 支持旧表升级：旧库缺少 `origin_client_id` 等列时先完成升级再创建依赖索引，避免自动同步初始化失败。
- 搜索索引维护入口改为 pending-only 增量构建：默认只处理待索引任务；全量重建保留为显式维护动作。
- 搜索索引新增 embedding profile 与文档 hash 元数据：可识别模型端点不一致、跳过 unchanged 文档，并清理 stale vector。
- Runtime 搜索索引面板增加稳定的运行进度 Inspector，展示 active/latest run、失败恢复和全量重建入口。
- 任务工作台增加状态 quick filter、摘要统计和更清晰的 metadata/tag 视觉层级，修复中文标签换行和密度问题。
- 数据库 migration 文件通过 `.gitattributes` 固定 LF，避免 Windows/跨平台换行导致 checksum 不一致。

### 修复

- 修复旧版远端 PostgreSQL sync schema 已存在但缺列时，基础 schema 初始化先创建依赖索引而失败的问题。
- 修复同一实体版本在不同客户端产生不同 payload/origin 时未能明确记录冲突的问题。
- 修复搜索索引操作默认全量重建过重、进度展示不稳定的问题。
- 修复任务工作台顶部筛选和详情元信息视觉语义混用的问题。

### 迁移与兼容性

- 本版本新增 SQLite migration：
  - `0009_sync_auto_conflicts.sql`
  - `0010_search_index_incremental_metadata.sql`
- `sync.auto.*` 默认关闭；升级后不会自动开始后台同步。
- 旧搜索索引库如果缺少 embedding profile，增量构建会进入需要全量重建的保护状态；先执行全量重建后再使用增量构建。
- 搜索向量库和派生 chunk 仍是本地派生状态，不作为同步权威数据。
- 已发布 migration 文件不要手动改换行或内容；跨平台换行一致性由 `.gitattributes` 维护。

## 版本口径

- Agenta 项目台账版本：`v0.1.3-release`。
- Manifest 版本源已通过 `bun run release -- --set-version 0.1.3` 对齐到 `0.1.3`：
  - `package.json`
  - `src-tauri/tauri.conf.json`
  - `src-tauri/Cargo.toml`
  - `src-tauri/Cargo.lock`
- 正式产物由 `bun run release` 写入 `target/release-artifacts/agenta-v0.1.3+<commit>`。

## 打包闸口

打包前至少执行：

```powershell
bun run build
cargo check --manifest-path src-tauri/Cargo.toml
cargo test --manifest-path src-tauri/Cargo.toml
```

建议额外关注：

```powershell
cargo test --manifest-path src-tauri/Cargo.toml --test sync_foundation -- --nocapture
cargo test --manifest-path src-tauri/Cargo.toml --test workspace_regression -- --nocapture
bun run release -- --dry-run
```

## 回滚提示

- 如果自动同步暂停，先查看 unresolved conflict，再决定保留哪一端的数据；不要直接清空远端 mutations。
- 如果搜索索引状态显示 embedding profile unknown 或 mismatch，优先执行全量重建。
- 如果遇到 migration checksum 错误，先确认工作区没有修改已发布 migration 文件，并备份用户数据库后再处理。
