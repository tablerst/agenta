# Agenta - 桌面端 UI 交互设计规范 (UI_DESIGN.md)

## 0. 文档定位 (Document Scope)

- 本文件是 Agenta 桌面端 UI 的**交互真相源**（Interaction Source of Truth）。
- 本文件负责定义：页面结构、区域职责、状态语义、键盘行为、弹层边界、阻塞级别和核心业务路径。
- 本文件**不负责**定义颜色值、阴影、圆角、字号、图标尺寸和视觉 token；这些由 `UI_STYLE.md` 负责。
- 若两份文档发生冲突：**交互结构与行为以本文件为准，视觉表达与 token 以 `UI_STYLE.md` 为准。**

## 1. 设计主旨 (Design Philosophy)

Agenta 作为一个面向开发者与 AI 协同的本地助手，其桌面端 (Tauri + Vue 3) 的设计核心是：**极简、命令式、高信息密度、键盘优先。**

整个界面应尽量减少“页面感”和“营销感”，转而像一个稳定、可靠、可快速切换上下文的工作台：

- 内容优先，而非装饰优先；
- 状态清晰，而非情绪化渲染；
- 键盘直达，而非鼠标路径依赖；
- 局部操作就地处理，而非强行全局打断。

## 2. 核心交互模型 (Core Interaction Model)

### 2.1 空间布局 (Spatial Layout)
采用经典的三栏/双栏结构，优先保证导航、对象列表与对象详情三类信息的稳定位置关系。

- **全局侧边栏 (Sidebar - 最左侧)**
    - 承担全局能力入口、全局动作入口、全局状态摘要。
    - 一级导航只保留真正的全局能力，例如 `Projects`、`Runtime`、全局搜索、主题、语言。
    - 侧边栏中的“搜索”“主题”“语言”等属于**全局工具**，不应与某个页面的局部表单混在一起。
    - 项目内的 `Tasks`、`Approvals`、`Versions` 不再作为一级导航项存在，而是收敛到项目工作区内部。
    - 当前路由的激活态应与列表选中态共享同一套语义：**轻量背景 + 细边框 + 文本抬升**。默认不使用 `3px` 彩色竖线作为主要选中语言。

- **项目工作区壳层 (Project Workspace Shell)**
    - 项目页不是单一表单页，而是一个稳定的父壳层：左侧为项目注册表，右侧为当前项目工作区。
    - 项目工作区头部必须固定展示：项目标题、摘要、状态，以及项目级二级导航（例如 `概览`、`版本`、`任务`、`审批`）。
    - 工作区头部下方的内容区负责承载项目级列表与详情，不允许因为长表单、长 JSON 或版本列表把整个页面顶出视口。
    - 项目切换时，工作区上下文必须整体切换；旧项目的列表选择、详情选择和审批详情不得残留到新项目中。

- **操作与列表栏 (List View - 中间)**
    - 承担当前路由上下文中的对象列表、局部筛选器、轻量创建入口、空态提示。
    - Hover 只用于预览和发现，不代表真正选中；Selected 才代表当前工作对象。
    - 当页面同时出现“筛选 + 创建 + 列表”时，必须保持视觉和语义分层，避免用户误以为所有区域都同权。

- **内容与详情栏 (Detail View - 右侧大区)**
    - 承担当前对象的摘要、结构化内容、日志、审批解释与附件信息。
    - 若当前没有选中对象，详情区必须显示明确空态，而不是留出语义不明的大片空白。

- **紧凑视口 (Compact Viewport)**
    - 视口收窄时，侧边栏应优先抽屉化或折叠化。
    - 不允许在窄视口中保留多层同时滚动的三栏挤压布局；要优先保住当前主要工作区的可读性。
    - 移动端可以将项目注册表、工作区列表、详情检查器改为纵向堆叠或抽屉化，但不能保留桌面端那种多层并发滚动压缩布局。

### 2.2 状态语义 (State Semantics)
为了避免实现时把 Hover、Selected、Focus 混为一谈，以下语义必须固定：

- **Hover**：表示“鼠标经过，可探索”，不改变当前上下文归属。
- **Selected**：表示“当前对象/当前条目已选中”，会决定详情区内容。
- **Active Route**：表示“当前路由处于激活状态”，本质上是导航层的 Selected。
- **Focus**：表示“键盘当前可操作位置”，只服务于键盘导航和可访问性。
- **Pressed / Active**：表示“当前正在按下或刚触发”的瞬时状态，不应与持久选中态混用。
- **Disabled**：表示“当前可见但不可操作”，必须保留结构信息，不能像被删除一样消失。

### 2.2.1 导航、Tab 与 Chip 语义边界

- **路由级二级导航**：用于项目工作区的 `概览 / 版本 / 任务 / 审批` 切换，本质上属于页面级导航，应该映射到路由，而不是页面内 tabpanel。
- **Tab 组件**：只用于同一详情对象内部的平级内容切换，例如任务详情中的 `概览 / 备注 / 附件 / 活动`。必须使用真正的 `tablist / tab / tabpanel` 语义，并支持键盘切换与焦点恢复。
- **Filter Chips**：只用于筛选状态或范围，例如审批状态、审批范围。Chip 不能冒充 Tab，也不承担 tabpanel 关系。
- **普通按钮**：用于即时动作，不得通过“看起来像 tab”来表达持久选中态。

### 2.3 弹层与阻塞模型 (Overlays & Blocking Model)
Agenta 的弹层体系必须有明确分工，不能把所有问题都塞进一个“看起来像浮层”的容器里。

- **Command Palette**
    - 用于全局搜索、全局跳转、全局命令执行。
    - 永远是**居中、键盘优先、非页面内嵌**的弹出式交互。
    - 打开后应自动聚焦输入区域；`Esc` 关闭；关闭后焦点返回触发源。
    - 不应用 Drawer 替代 Command Palette。

- **Drawer**
    - 用于当前上下文内的多步骤局部操作，例如查看补充信息、填写局部表单、处理某个对象的次级流程。
    - Drawer 不承担全局搜索与全局导航职责。
    - Drawer 打开时，用户必须仍能感知背后的主上下文存在。

- **Modal**
    - 只用于阻断性、高风险或必须立即确认的场景，例如严重错误、不可逆破坏性确认、系统级脱机。
    - Modal 会提升阻塞级别，应谨慎使用。

- **Popover / Menu / Flyout**
    - 用于局部选项选择、轻量设置和短操作列表。
    - 若内容超过简单选项集合，不应继续塞在 Popover 中，应升级为 Drawer 或独立面板。

- **Toast / Ephemeral Feedback**
    - 用于轻量反馈，如成功、已复制、已同步。
    - 不承担关键路径决策，也不应成为唯一的错误承载方式。

### 2.4 核心业务交互路径 (Core Flows)

#### A. 全局搜索 (Global Search - FTS5)
- **触发方式**：全局快捷键 (`Cmd/Ctrl + K`) 与侧边栏中的搜索触发器。
- **语义约束**：侧边栏搜索入口即使外观借用了输入框 affordance，本质仍是一个触发 Command Palette 的按钮，而不是页面内真实输入框。
- **行为约束**：Palette 打开后立即可键入，支持上下选择、回车执行、`Esc` 关闭。
- **结果定位**：搜索结果应直接映射到“跳转 / 打开 / 聚焦某对象”的动作，而不是先让用户进入第二层页面再继续找。

#### B. 任务、日志与 Payload (Tasks & Logs)
- 任务页默认绑定当前项目，不再提供“全部项目”筛选；任务筛选仅保留版本和状态。
- 列表区负责对象筛选、创建和切换；详情区负责解释与展开。
- 结构化日志、JSON Payload、附件纯文本必须使用 Pretty 结构化展示，不能退回高噪声终端样式。
- “无选择”与“无数据”必须区分：前者是用户尚未选中对象，后者是当前筛选条件下没有结果。
- 任务详情中的 `概览 / 备注 / 附件 / 活动` 必须实现为真正的 Tab 组件，而不是一组视觉上类似 Tab 的普通按钮。

#### C. AI 拦截与审批流 (Policy Approvals)
- **触发场景**：策略动作命中 `require_human` 或 `deny` 等规则时。
- **入口层级**：全局导航中可以显示待审批数量，但审批决策本身必须落回项目工作区内的审批区段、详情区或 Drawer，而不是仅靠一个 badge 解决。
- **信息组织**：审批界面需要同时提供风险说明、Diff、原始指令/摘要、人类决策入口。
- **动作优先级**：`Approve` 与 `Deny` 的视觉与交互优先级必须有区分，但不能演变为强情绪化大红大绿按钮战争。
- 审批区段默认显示当前项目的审批请求，并提供显式的“全部审批”范围切换。
- 审批响应必须直接给出项目归属与任务引用；前端不负责根据 `action + resource_ref` 猜测项目上下文。
- 审批状态与范围切换使用 Filter Chips，而不是 Tab 语义；审批详情中的“跳转到资源”必须回到对应项目的版本或任务区段。

#### D. 系统级异常与离线状态 (System Warnings & Offline)
- 系统级异常（如数据库脱机、桥接失败、权限缺失）需要明确区分为：
    - **可继续工作**：用局部提示或 Toast；
    - **当前流程受阻**：用就地错误面板或 Drawer 提示；
    - **全局不可继续**：才允许上升到 Modal。

## 3. 键盘优先、动效与焦点恢复 (Keyboard, Motion & Focus Restoration)

- **键盘优先 (Keyboard First)**：所有高频路径必须可通过键盘完成，尤其是全局搜索、列表切换、审批与关闭弹层。
- **焦点恢复 (Focus Restoration)**：关闭 Palette、Drawer、Modal 后，焦点应返回触发源或最近的稳定上下文，不允许用户“键盘迷路”。
- **动效原则**：过渡应短促、利落、可预测，不引入情绪化的回弹。
- **Reduced Motion**：若系统或应用启用减少动态效果，应保留层级变化，但减少 spotlight、位移动画和渐变扫光。

## 4. 空态、加载态与错误态策略 (Edge-State Strategy)

- **空态 (Empty State)**：用简短说明明确“为什么这里为空、下一步能做什么”。
- **无结果 (No Results)**：与空态区分开来，应强调“当前筛选条件导致没有结果”。
- **加载态 (Loading)**：优先 Skeleton，并保持布局稳定，避免内容区来回跳动。
- **禁用态 (Disabled)**：禁用是暂时不可操作，不是视觉删除；需要保留标签、结构和解释。
- **错误态 (Error State)**：优先就地解释问题，并在必要时提供恢复动作，不要只丢一句失败提示。

## 5. 维护与变更守则 (Governance)

- 新增页面、路由、弹层类型或交互路径前，必须先判断它属于 Sidebar、List、Detail、Palette、Drawer、Modal 中的哪一类。
- 若某次改动引入了新的交互模式（例如新的审批流程入口、新的多步编辑模式），必须先更新本文件再实现。
- 若某次改动同时改变了交互结构和视觉外观，必须在同一个变更中同步更新 `UI_DESIGN.md` 与 `UI_STYLE.md`。
- 任何 PR 若新增了新的页面模式、overlay 模式或状态语义，应在描述中明确引用所更新的规范章节，避免“代码先漂、文档后补”。
