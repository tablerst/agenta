# Agenta - 桌面端视觉指引 (UI_STYLE.md)

## 0. 文档定位 (Document Scope)

- 本文件是 Agenta 桌面端 UI 的**视觉真相源**（Visual Source of Truth）。
- 本文件负责定义：颜色、透明度、圆角、阴影、字号、密度、图标、分割线、滚动条、状态视觉、组件外观和动效数值。
- 本文件**不负责**定义业务流程、页面职责、弹层使用场景、键盘行为和阻塞级别；这些由 `UI_DESIGN.md` 负责。
- 若两份文档发生冲突：**交互结构与行为以 `UI_DESIGN.md` 为准，视觉表达与 token 以本文件为准。**

## 1. 视觉主题 (Visual Identity)

Agenta 是一款以 CLI 和 AI 协作为核心的现代化工具。其视觉语言深度借鉴 **Linear 的黑白极简美学**，呈现出“克制、锐利、专业、高信息密度”的质感。

### 1.1 主题切换与光感基调 (Theme & Illumination)
- **双主题无缝支持 (Light / Dark Mode)**：原生支持系统级明暗模式切换。无论是在深邃的暗色环境中，还是在通透的亮色背景下，都必须保持一致的密度感、秩序感和高级光影质感。
- **色彩与光影策略 (Light over Color)**：**极致去色，以光代色**。层级与反馈不依赖大面积纯色块，而依赖极弱的高光边、低透明度灰阶填充、细微的流光和指针聚光灯跟随。
- **主基调 (Monochrome First)**：界面以黑、白、灰为主体，彩色只允许作为状态提示和极小面积的结构噪声，不能反客为主。

## 2. 视觉 Token 规范 (Visual Tokens)

### 2.1 颜色、材质与光影 (Colors & Lighting)
采用 Linear 风格的高对比黑白灰调色板，并通过极低透明度的光影关系塑造层级。

*   **背景色与层级光感 (Backgrounds)**
    *   主视图基础底色 (Main Background): `Light: #FFFFFF` / `Dark: #0A0A0A`
    *   侧边栏与悬浮层 (Elevated Surfaces): `Light: #F7F7F8` / `Dark: #141415`
    *   高一级容器/浮层 (Elevated Panels): 允许在主底色基础上通过 `2% ~ 8%` 的明暗差异来区分，不允许跳出灰阶体系。
    *   光感线与边框 (Glowing Borders): `Light: rgba(0,0,0,0.04)` / `Dark: rgba(255,255,255,0.06)`。默认使用 `1px` 边界线，不允许用粗重描边或大面积投影替代边界。
    *   交互悬浮 (Interactive Hover & Spotlight): 鼠标悬浮时允许使用极弱径向渐变（Radial Gradient）跟随指针，但透明度必须保持克制，不能覆盖内容本身。
*   **文本色 (Text & Content)**
    *   主标题/关键信息 (High Contrast): `Light: #0F0F0F` / `Dark: #EDEDED`
    *   次级说明/元信息 (Medium Contrast): `Light: #737373` / `Dark: #A1A1AA`
    *   占位文本 (Placeholder): 只能在 `text-muted` 基础上再下降一个层级，仍需可辨识，严禁低对比到“看不见”。
*   **主强调与状态色 (Accent & Status States)**
    *   主强调 (Primary Accent): 采用**黑白反转**，只用于主行动点、品牌识别、小范围聚焦提示和键盘 focus 系统，不用于大面积页面底色。
    *   主强调渐变 (Accent Gradient): 仅允许在同一灰阶体系内做弱渐变，亮度差建议控制在 `5% ~ 12%`。
    *   `allow`：仅用于小型 badge、图标点缀、细边框或轻微背景染色；背景透明度上限建议为 `0.10`。
    *   `require_human`：仅用于审批提示、待处理 badge、细边框和低频率脉冲式提示；背景透明度上限建议为 `0.12`。
    *   `deny`：仅用于阻断性提示、风险 badge、细边框和图标；背景透明度上限建议为 `0.10`，禁止整张卡片铺满红色。
    *   **状态色禁令**：`allow / require_human / deny` 不得用于大面积页面主背景、整块面板底色或常态文本主色。
*   **代码高亮配色 (Syntax Highlighting - 专属克制版)**
    *   禁止使用默认高饱和终端配色。
    *   `Dark`: Key 偏灰蓝、String 偏灰绿、Number 偏灰桔、Punctuation 偏中性灰。
    *   `Light`: 使用相同逻辑的低饱和对应色，避免出现“彩色 IDE 主题”感。

### 2.2 排版与字体 (Typography)
因为涉及大量文本摘要、日志记录和代码，字体与密度必须保持冷静、紧凑、可扫读。

*   **界面主字体 (UI Font)**: 首选系统原生无衬线字体，如 `Inter`, `San Francisco`, `Segoe UI`, `Roboto`。
*   **代码与终端字体 (Monospace Font)**: 凡涉及 MCP Payload、附件纯文本、日志、摘要字段内容，强制使用等宽字体，如 `JetBrains Mono`, `Fira Code`, `SF Mono`。
*   **字阶规则 (Type Scale)**
    *   H1 / H2：`20px ~ 24px`，字重 `600`
    *   分区标题 / 面板标题：`14px ~ 16px`，字重 `600`
    *   正文：`13px ~ 14px`，字重 `400 ~ 500`，行高约 `1.5`
    *   元信息（时间戳、版本号、快捷键提示）：`11px ~ 12px`，字重 `400`
    *   标签与字段名：建议使用 `11px ~ 12px` 的小号字配合较大的字间距，体现仪表盘式秩序感。

### 2.3 几何、密度与基础元素 (Geometry, Density & Primitives)
*   **圆角 (Border Radius)**
    *   **严禁使用大圆角**：禁止使用 `rounded-xl`、`rounded-2xl` (`16px`) 或更大圆角来制造“果冻感”。
    *   **基础控件**：按钮、输入框、导航槽、列表项应使用 `4px ~ 6px` 圆角。
    *   **中型容器**：分区面板、详情块、表单卡片应使用 `8px ~ 12px` 圆角。
    *   **禁止例外泛滥**：若某一组件想突破上限，必须先更新本文件，而不是在实现中偷偷放大。
*   **高度与密度 (Heights & Density)**
    *   紧凑控件（搜索触发、次级按钮、图标按钮、筛选器）高度：`32px`
    *   标准表单控件（输入框、下拉框、主按钮）高度：`36px`
    *   导航项/列表行：`36px ~ 40px`
    *   Badge/状态胶囊：`20px ~ 24px`
    *   桌面端不应默认采用 `48px` 级高度；如确有可触达需求，必须是明确的响应式例外。
*   **图标 (Icons)**
    *   默认使用无填充 Outline 图标。
    *   常规图标尺寸：`14px ~ 16px`；导航或重点入口允许 `16px ~ 18px`。
    *   图标应与文字光学居中对齐，不能因为图标过大导致控件显得肿胀。
*   **分割线与边界 (Dividers & Edges)**
    *   所有分割线默认 `1px`。
    *   分割线优先通过透明度和灰阶层级表达，不通过阴影假装分割线。
*   **空间网格 (Grid & Spacing)**
    *   基于 `4px / 8px` 网格系统。
    *   面板和区域内边距应优先保证文字扫描效率，而不是追求“松弛的大留白”。

### 2.4 状态矩阵与动效数值 (State Matrix & Motion Tokens)
*   **默认态 (Default)**：以边框、文本对比度和轻微灰阶差作为主体，不抢内容。
*   **悬浮态 (Hover)**：只能提升 `1` 个视觉层级，例如略微增强边框、提升表面亮度或出现极弱 spotlight；Hover **不是** Selected。
*   **选中态 (Selected)**：使用轻微底色、`1px` 边框和文本对比度抬升表达；不能依靠高饱和色条作为唯一信息。
*   **聚焦态 (Focus Visible)**：使用 `1px ~ 2px` 的弱 focus ring 或边框强调；Focus 用于键盘定位，不等同于 Hover 或 Selected。
*   **按压态 (Active / Pressed)**：只允许 `1px` 级别的压缩感、阴影收紧或填充微调；禁止“深凹陷、大内阴影、厚重机械键帽”式处理。
*   **禁用态 (Disabled)**：透明度降低、对比度下降、取消发光反馈；不得通过纯灰蒙版把内容彻底洗掉。
*   **加载态 (Loading)**：优先采用 Skeleton，占位骨架应保持单色和低对比；非必要不使用强存在感 Spinner。
*   **空态 (Empty)**：优先文字解释 + 轻量图标，不依赖插画填满空间。
*   **动效时长 (Motion Durations)**
    *   Hover / Focus / Spotlight：`100ms ~ 150ms`
    *   Panel / Drawer / Palette 打开与关闭：`180ms ~ 220ms`
    *   禁止使用弹簧、果冻、回弹或任何夸张 overshoot 动效。

## 3. 组件视觉基准 (Component Baselines)

### 3.1 表单与输入区 (Inputs & Forms)
- **拒绝深凹与内阴影 (No Sunken Insets)**：输入框、搜索框和文本域禁止使用深 `inset` 阴影制造下陷感。
- **输入暗示的来源**：依靠 `3% ~ 6%` 的灰阶差、`1px` 边框、合理的 placeholder 对比度和稳定的高度建立“可输入”感，而不是靠凹槽感。
- **搜索触发器规则**：外观可以借用输入框的克制感，但若语义上是按钮，则必须保留按钮交互反馈，不得伪装成真的可编辑输入域。
- **字段一致性**：输入框、下拉框、搜索触发器和小型过滤器必须共用同一套圆角、高度和边界语法，避免一页之内出现多种输入气质。

### 3.2 按钮与控件 (Controls & Actions)
- **次级按钮 (Secondary)**：以中性灰阶底色、细边框和极弱顶边高光为主，不应过于肥厚。
- **主按钮 (Primary)**：采用黑白反转体系，可保留极弱的顶部高光和很短的底部阴影，但总体应干净、压缩、利落。
- **图标按钮 (Icon Button)**：保持正方形比例，小半径、弱边框、小阴影，不允许做成圆润泡泡按钮。
- **危险动作 (Danger)**：默认不是整块红按钮；只有在明确 destructive confirm 场景下，才允许提升状态色存在感。

### 3.3 导航、列表与选择高亮 (Navigation, Lists & Selection)
- **统一选中语言**：导航项、列表项、过滤 chip、命令面板条目应尽可能共享同一套 Selected 视觉语法：轻微底色 + 细边框 + 文本抬升。
- **禁止高饱和竖条主导激活态**：左侧强调线如果存在，只能作为辅助，而不能成为主视觉语言；默认推荐不使用 `3px` 彩色竖条作为主要激活手段。
- **Hover 与 Selected 分离**：Hover 只提供预览感，Selected 才表示“当前上下文归属”。

### 3.4 徽章、卡片与状态面板 (Badges, Cards & Status Panels)
- **Badge 体量要小**：Badge 只承担状态提示，不承担整块面板着色任务。
- **策略与审批卡片**：允许低浊底色、弱渐变和顶边高光，但风险信息应由结构和文案主导，而不是靠大片红黄底色恐吓用户。
- **状态映射**：`allow / require_human / deny` 必须在 badge、图标、细边框等小面积元素中体现，避免在整页中扩散。

### 3.5 弹层与覆盖物外观 (Overlays & Elevation)
- **Command Palette**：视觉上最中性，边界最清晰，强调“工具”而非“对话框”。
- **Drawer**：边缘附着感更强，应与主内容区有明确的 `1px` 边界分隔。
- **Modal**：遮罩最重、层级最高，但面板本体仍保持单色克制，不做炫目的警告光效。
- **Popover / Menu / Flyout**：尺寸更紧凑，边框与表面关系应与主界面一致，不能突然换成另一套审美。

### 3.6 空态、加载态、禁用态、滚动条与代码块 (Edge Surfaces)
- **空态 (Empty State)**：允许使用单色图标和短说明，但不能出现营销式插画或大面积品牌色。
- **加载态 (Loading State)**：优先 Skeleton，其亮度变化必须温和，不得闪烁。
- **禁用态 (Disabled State)**：仅降低可操作性，不隐藏信息结构。
- **滚动条 (Scrollbar)**：应细、轻、低存在感；Hover 时略提升对比，不得成为大面积装饰元素。
- **代码块 (Code Blocks)**：必须延续整体灰阶体系，低饱和、低噪声、可读优先。

## 4. 视觉反模式 (Anti-patterns)

- **拒绝大圆角、深内阴影与泡泡控件**：任何会把工具界面做成“圆润 App 卡片”的处理都不符合本产品气质。
- **拒绝炫彩主导与品牌色泛滥**：禁止把蓝紫、霓虹红绿、彩色渐变当成默认交互反馈。
- **拒绝状态语义混乱**：Hover、Selected、Focus、Active 不能长得几乎一样，更不能靠同一种彩色底来糊过去。
- **拒绝大面积状态染色**：风险、审批、允许等状态必须克制地附着在局部元素上，而不是整块染色。
- **拒绝拖沓动效**：禁止回弹、拖尾、果冻、物理惯性过重的过渡。

## 5. 维护约束 (Governance)

- 新增组件必须先映射到现有 token、状态矩阵和组件基线，**不能先写实现再事后补文档**。
- 若要引入新的颜色、圆角、高度、阴影或状态视觉，必须先更新本文件，再落地实现。
- 任何新页面若引入新的视觉模式（例如新的卡片语言、新的 overlay 皮肤），必须同时说明为何不能复用现有模式。
